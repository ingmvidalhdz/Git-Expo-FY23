Comandos basicos

  • git pull: Para bajar o descargar cambios de la rama en la que nos encontramos situados

  • git pull origin "nombre-rama": Para bajar o descargar cambios de alguna rama activa o diferente (main)

  • git add .: Para agregar todos los cambios

  • git add ruta_nombre_archivo: Para agregar un cambios especifico en un archivo

  • git commit -m "mensaje": Para agregar un breve mensaje sobre los cambios realizados.

  • git push origin -u "rama a la cual se enviaran": Para enviar cambios, la rama tiene que ser sobre en la que estamos situados o en la que estamos trabajando

  • git status: Para ver el status de nuestro repositorio, si tenemos cambios sin subir o si esta desactualizado nuestro repositorio

  • git log: Para ver el historial de commits

  • git stash: Para "esconder" tus cambios

  • git branch: Para mostrar las ramas de nuestro repositorio

  • git checkout -b "NombreRama": Para crear una nueva rama

  • git checkout NombreRama: Para cambiarse a una rama en concreto

Branches (Ramas)

  • Las ramas en git, son una especie de camino en el cual se concentraran los cambios de cada caractersitica a desarrollar.

  • Obligatoriamente se tendra al menos una rama base (main / master), esto depende del equipo de desarrollo y se pueden tener hasta 3 ramas base

    • Develop
    • QA
    • Producción

  • Las ramas son creadas por cada desarollador

  • Se tiene que respetar cierta nomenclatura para nombrar nuestras ramas (esto depende del equipo de desarollo)

    • Para agregar una nueva caracteristica: "feature/#num_NombreCaracteristica". Ejemplo: feature/#12_RegistroUsuarios
    • Para solucionar un bug con prioridad moderada o baja: "bugfix/#num_NombreBug
    • Para solucionar un bug con prioridad alta: "hotfix/#num_NombreBug"
    • Para el entregable o versió con avances para mostrar al usuario: "release/01"

Merge

+El merge basicamente es la fusion de cambios entre 2 ramas.

+Si aplicamos un Merge, solo se encontarian los caminos del registro de cambios de nuestras ramas

+Por ejemplo si estamos en una rama con una caracteristica nueva y queremos unir los cambos de otra rama activa. Debemos jalar o traer la rama activa.

Ejemplo

  • Primero nos situamos en la rama en la cual queremos traer los cambios: git checkout feature/#23_LoginFrontend
  • Despues identificamos la rama de la cual queremos traer los cambios en este caso la rama que jalaermos sera: feature/#54_ApiLogin
  • Una vez identificadas las ramas, procedemos a hacer el merge con el comando: git merge feature/#54_ApiLogin
  • Si surgen conflictos tendremos que resolverlos, en caso de que no surja ninguno, la rama sobre la que estamos pocisionados (feature/#23_LoginFrontend) jalara los cambios de la rama activa (feature/#54_ApiLogin)

+Un ejemplo más simple es cuando se hace un PR (Pull Request), github por detras hace un Merge, jalando los cambios de nuestra rama (feature) dentro de la rama base (main / master)

Rebase

+El rebase hace algo similr al merge, pero con la excepción de que por detras hace la unión de 2 ramas en 1 sola

+Si aplicamos un rebase, el camino o registro de cambios de la rama en la que estamos trabajando se perdera una vez hecho el PR

+Por ejemplo si estamos en una rama con una caracteristica nueva a desarrollar y queremos intergrarla con la rama base (master o main), debemos hacer lo siguiente

  • Primero nos situamos en la rama en la cual queremos traer los cambios: git checkout feature/#23_LoginFrontend
  • Despues identificamos la rama de la cual queremos traer los cambios en este caso la rama que jalaermos sera: main
  • Una vez identificadas las ramas, procedemos a hacer el merge con el comando: git rebase main
  • Si surgen conflictos tendremos que resolverlos, en caso de que no surja ninguno, la rama sobre la que estamos pocisionados (feature/#23_LoginFrontend) jalara los cambios de la rama activa (main)
  • Para el caso de que surjan conflictos, debermos resolver conflictos y hacer un commit, una vez hecho esto continuamos con git rebasse --continue

Conflicts

-Son problemas causados, principalmente cuando 2 desarolladores o más estan editando los mismos archivo o directorios

-Estos conflictos forsozamente se deben resolver de manera manual

-Por lo regular se hacen presentes cuando hacemos un pull, rebase o merge

Cherry Pick

-Basicamente es una tecnica mediante la cual, podemos traer commits especificos desde otra rama, pero sin la necesidad de unir sus cambios o fusionar ramas en 1 sola

-Es usado amenudo cuando surjen errores en ramas base alternas como develop o producción

-Es usada en casos muy especificos, pero igual depende del equipo de desarrollo
